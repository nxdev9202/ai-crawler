# 07. 3계층 — LLM 해석 설계 (Claude Opus 4.8)

> 결정 사항: 해석 모델은 **Claude Opus 4.8** (`claude-opus-4-8`). 신뢰도·추론력 우선 방침 확정.

## 모델 스펙 & 비용 (Claude API 공식 레퍼런스 기준, 2026-06 캐시)

| 항목 | 값 |
|---|---|
| 모델 ID | `claude-opus-4-8` (정확히 이 문자열, 날짜 suffix 붙이지 말 것) |
| 컨텍스트 | 1M 토큰 / 최대 출력 128K |
| 가격 | 입력 **$5.00/1M tok**, 출력 **$25.00/1M tok** |
| thinking | `thinking: {type: "adaptive"}` — **budget_tokens는 400 에러** (구식 문법 금지) |
| 샘플링 | **temperature/top_p/top_k 파라미터 자체가 제거됨 (보내면 400)**. 톤·다양성은 프롬프트로 제어 |
| effort | `output_config: {effort: "high"}` 권장 (해석 품질 우선; medium으로 비용 절감 가능) |
| 스트리밍 | 긴 출력이므로 **필수** — SDK `client.messages.stream()` + `finalMessage()` |
| 프리필 | assistant 프리필 금지(400). JSON이 필요하면 `output_config.format` (structured outputs) |

### 요청 1건당 예상 비용 (전체 풀이 기준)

- 입력: 시스템 프롬프트(~3K) + 명식(~1K) ≈ 4K tok → $0.02
- 출력: 전체 풀이 ~8K tok → $0.20
- **건당 약 $0.22 (약 300원).** 판매가 2.9~3.9만원 대비 원가율 ~1% — tosoksaju 모델 그대로 성립.
- 시스템 프롬프트에 `cache_control: {type: "ephemeral"}` 걸면 입력측 90% 절감 (트래픽 몰릴 때 유효).

## 아키텍처 원칙 (정확도의 핵심)

**"계산은 코드, 해석은 LLM" — LLM에게 절대 사주를 계산시키지 않는다.**

- LLM은 만세력 DB가 없고 텍스트 패턴만 학습했으므로 생년월일→팔자 계산을 시키면 팔자 자체를 틀린다.
- ssaju가 뽑은 명식(원국·십성·격국·용신·대운, ~950토큰)을 **구조화된 사실**로 주입하고, 해석·서사만 맡긴다.
- 이것이 "체감 정확도"를 올리는 가장 큰 단일 요인이다. 프롬프트 기교보다 우선.

## 프롬프트 구조

```
[system — cache_control 적용, 절대 바이트 고정]
 1. 역할 정의 (명리 해석가 페르소나)
 2. 해석 규칙 (아래 '정확도 규칙')
 3. 출력 구조 (섹션 템플릿)
 4. 문체/톤 지침 (페르소나 말투)
[user]
 5. 명식 데이터 (ssaju 출력 그대로)
 6. 사용자 컨텍스트 (성별, 혼인여부, 시간모름 여부)
 7. 요청 범위 (미리보기 or 전체)
```

## 시스템 프롬프트 초안

```
당신은 자평명리학에 정통한 사주 해석가다.

## 절대 규칙
1. 아래 제공되는 명식 데이터(원국, 십성, 격국, 용신, 대운 등)만을 사실로 사용한다.
   생년월일로 간지를 다시 계산하거나, 제공되지 않은 신살·격국을 임의로 추가하지 않는다.
2. 해석은 음양오행 생극제화, 십성의 심리·사회적 의미, 격국과 용신, 대운·세운의
   희기(喜忌) 판단이라는 명리학의 표준 논리 사슬로 전개한다.
   각 판단은 "명식의 어떤 글자 때문인지"를 근거로 제시한다 (예: "월지 유금이 일간 갑목을 극하니…").
3. 단정적 예언("몇 년에 죽는다", "이혼한다")은 하지 않는다. 건강·법률·투자에 대한
   구체적 지시 대신 경향과 주의 시기를 말한다.
4. 시간 미상(시주 없음)인 경우 삼주만으로 해석하고, 시주 관련 단정을 피한다고 명시한다.
5. 모든 한자 용어는 한글(한자) 병기 첫 1회 후 한글로 쓴다.

## 출력 구조
1. 총평 — 원국의 핵심 구도 (일간 강약, 격국, 용신) 3~4문장
2. 타고난 성정 — 십성 분포 근거
3. 재물운 — 재성의 상태와 대운 흐름
4. 애정/인연운 — 관성·재성(성별에 따라), 배우자궁(일지)
5. 건강 — 오행 과불급 기준 (의학적 진단 아님을 명시)
6. 대운 흐름 — 현재 대운과 향후 2개 대운의 희기
7. 올해와 내년 — 세운 간지와 원국의 합충 관계
8. 조언 — 용신 기준의 실천적 개운 방향

## 문체
[페르소나 지침 — 서비스 캐릭터 확정 후 삽입.
 예: 경상도 사투리를 쓰는 노년 여성 역술가. 반말, 직설적이되 따뜻함.
 "~했제", "~인기라" 등. 과도한 사투리로 가독성을 해치지 않는다.]
```

## 정확도(체감 포함)를 올리는 기법 — 조사 결과 정리

1. **계산/해석 분리** (위 원칙) — 가장 효과 큼.
2. **근거 강제**: "명식의 어떤 글자 때문인지 인용하라" → 콜드리딩식 두루뭉술 문장이 줄고 개인화 체감 상승.
   (smeuse 블로그 사례: 근거 인용 강제만으로 체감 정확도 대폭 상승 보고)
3. **구조화 입력**: 십성 개수, 오행 분포 점수 등 수치화된 데이터를 주면 "비겁 3개로 강한 일간" 같은
   정량 근거 문장이 나온다. ssaju 출력이 이미 이 형태.
4. **금지 목록 명시**: 재계산 금지, 미제공 신살 창작 금지, 단정 예언 금지 — Opus 4.8은 지시를 문자 그대로
   잘 따르므로 금지 규칙의 효과가 크다.
5. **사용자 피드백 루프 대신 컨텍스트 주입**: 혼인 여부·성별을 받는 이유(tosoksaju도 동일)는
   관성/재성 해석 방향을 정하기 위함. 반드시 프롬프트에 전달.
6. **미리보기/전체 분리**: 미리보기는 총평+성정만 생성(출력 ~1.5K tok, 건당 ~$0.06),
   결제 후 전체 생성. 미리보기에 "가장 놀라운 한 가지"를 넣으면 전환 훅이 된다.
7. **온도 파라미터가 없으므로** 문체 다양성은 프롬프트로: "같은 표현을 반복하지 말라" 등.

## 호출 코드 스켈레톤 (TypeScript, Next.js API 라우트)

```typescript
import Anthropic from "@anthropic-ai/sdk";
const client = new Anthropic();

const stream = client.messages.stream({
  model: "claude-opus-4-8",
  max_tokens: 16000,
  thinking: { type: "adaptive" },
  output_config: { effort: "high" },
  system: [{ type: "text", text: SYSTEM_PROMPT, cache_control: { type: "ephemeral" } }],
  messages: [{ role: "user", content: buildUserPrompt(myeongsik, userContext, scope) }],
});
const message = await stream.finalMessage();
```

- `SYSTEM_PROMPT`는 바이트 단위로 고정할 것 (타임스탬프·유저ID 삽입 금지 — 캐시 무효화됨).
- 에러 처리: `RateLimitError`(재시도), `APIStatusError` 구분. 결제 후 생성 실패 시 재시도 큐 필수.

## 평가 계획 (개발 단계)

- 골든 명식 10개(경계 케이스 포함)에 대해 생성 결과를 체크리스트 평가:
  (a) 명식에 없는 사실 창작 여부 (b) 근거 인용률 (c) 섹션 구조 준수 (d) 페르소나 일관성
- 동일 명식 3회 생성해 핵심 판단(신강/신약, 용신)이 흔들리지 않는지 확인 — 흔들리면 명식 데이터에 판단을 포함시켜 고정.
```
