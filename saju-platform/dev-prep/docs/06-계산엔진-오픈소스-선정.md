## ssaju 채택 근거

1. **2계층 요구사항을 한 번에 충족**: 십성·격국·용신·대운·세운까지 별도 규칙 테이블 구현 없이 제공.
   다른 두 라이브러리는 만세력(1계층)까지만 커버해서 파생 계산을 직접 짜야 함.
2. **LLM-friendly 출력이 설계 목표**: 명식을 ~950토큰으로 압축해 그대로 Opus 4.8 프롬프트에 넣을 수 있음.
   3계층(LLM 해석)과의 인터페이스 비용이 사실상 0.
3. **스택 일치**: 서비스는 Next.js(TypeScript)로 갈 예정 (tosoksaju 분석 참조) → 서버 API 라우트에서 그대로 import.
4. 절입 시/분 기준 계산 → 본 프로젝트의 절입시각 데이터와 상호 검증 가능.
5. MIT 라이선스, 의존성 0개, 활발한 유지보수.

## 리스크와 보완책

| 리스크 | 보완 |

| 업스트림 중단 | MIT라 포크 가능. 버전 고정(lockfile) + 골든 테스트 스냅샷 유지 |

## 통합 설계

```
[사용자 입력] 생년월일·시간(모름 허용)·양/음력·윤달·성별·출생도시
      ↓
[전처리 래퍼] (우리 구현)
  - 음력 입력 → 양력 변환 (ssaju 또는 KASI DB)
  - 서머타임 −1h (korea_dst_periods.json)
  - 시대별 표준시 확인 (korea_timezone_history.json)
  - 진태양시 보정 (04 문서, 도시→경도 테이블)
  - 야자시 모드에 따른 날짜 귀속 결정 (05 문서)
      ↓
[ssaju] 명식 계산 → 원국/십성/12운성/신살/격국/용신/대운/세운 + LLM용 압축 출력
      ↓
[검증 레이어] (개발 중에만) KASI DB·자체 절기 데이터와 대조
      ↓
[3계층 LLM] Opus 4.8 해석 (→ 07 문서)
```

## 개발 착수 시 첫 작업

```bash
npm i ssaju
# 골든 테스트: 아래 케이스를 시중 만세력(만세력닷컴 등) 결과와 대조
# - 평범한 케이스: 1990-03-15 15:05 서울
# - 입춘 경계: 2024-02-04 17:20 / 17:35
# - 서머타임+구표준시: 1955-08-08 16:20
# - 야자시: 1990-03-15 23:45
# - 음력 윤달 입력 케이스
```
